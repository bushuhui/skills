# 高效编程协作模式：基于文档索引的迭代开发

## 核心理念

不要让 AI 每次都从头扫描整个代码库。通过**维护一份增量更新的文档索引**，AI 可以在需要时只读取已有的总结文档，快速理解项目全貌，避免盲目分析造成的时间浪费和信息遗漏。

### 工作流程

```
初始分析 ──> 生成 docs/ 文档 ──> 每次改动增量更新 ──> 新任务时指定文档路径让 AI 读取
```

---

## 第一步：项目初始分析

首次进入一个项目时，让 AI 全面扫描代码结构并生成基础文档。

### 提示词：项目架构分析

```
请分析当前项目的整体架构，在 docs/ 目录下生成以下文档：

1. docs/architecture.md — 项目架构总览
   - 技术栈与核心依赖
   - 目录结构（树状图，标注核心/配置/资源目录）
   - 模块划分及各模块职责
   - 模块间的依赖关系和数据流向

2. docs/api.md — 接口定义汇总
   - 所有 API 端点（路径、方法、参数、返回值）
   - 路由定义位置
   - 前后端交互方式

3. docs/changelog.md — 改动日志（初始为空模板，格式见下方）

文档要求：
- 基于实际代码分析，不猜测
- 无法确定的部分注明"待进一步分析"
- 使用清晰的 Markdown 格式，方便后续增量更新
```

---

## 第二步：每次改动后增量更新

在对话式协作中，代码改动通常是这样的节奏：**Claude 规划方案 → 按规划逐步修改 → 你运行验证后觉得没问题**。这时候只需要一句话让 Claude 把这次改动记录下来，不需要重新扫描代码。

### 日常小改动：追加 changelog

当你验证完改动、确认运行正常后，直接说：

```
非常好，把这次改动说明添加到 docs/changelog.md 里。
```

Claude 会根据对话中已经做过的改动，自动整理出变更记录并追加到 changelog 中。格式类似：

```markdown
## YYYY-MM-DD — 改动摘要
- **范围**：涉及的模块
- **修改**：file_a.ts, file_b.py
- **接口变更**：如有则列出
- **备注**：破坏性变更、待办事项等
```

### 大模块更新：新建独立文档 + 更新 changelog

如果是新增功能模块、重构大组件等较大变动，除了 changelog 记录，还应该为新模块建立独立说明文档。这时候说：

```
非常好，把本次更新的模块添加到新增 md 文件中。
```

Claude 会：

1. 在 `docs/` 下新建 `docs/modules/[module-name].md`，包含：
   - 模块职责和设计思路
   - 核心文件清单
   - 对外暴露的接口/函数/类
   - 与其他模块的依赖关系
   - 使用示例（如适用）

2. 更新 `docs/architecture.md`，在模块划分章节加入此模块

3. 在 `docs/changelog.md` 中追加一条大模块变动记录

### 关键原则

- **利用对话上下文**：Claude 已经知道刚才做了什么改动，不需要重新分析代码
- **先验证再记录**：你运行确认没问题之后再让 Claude 写文档，避免记录半成品
- **一句话触发**：不需要写长提示词，简单说"把这次改动加到 changelog"就行

## 第三步：新任务时指定文档路径

开启新开发任务时，**不要**让 AI 盲目扫描代码，而是直接指定已有的文档路径让它快速建立上下文。

### 提示词：新任务启动（带文档索引）

```
准备开始 [任务描述]。

请先阅读以下已有文档，建立对项目的理解：
- docs/architecture.md — 了解整体架构和模块划分
- docs/api.md — 了解接口定义
- docs/[相关模块].md — 了解涉及模块的细节
- docs/changelog.md — 了解最近的改动历史

基于这些信息，请：
1. 简要总结你理解的项目当前状态
2. 分析要完成这个任务需要修改哪些模块
3. 给出实现方案概要

如果已有文档的信息不够用，再针对性地扫描相关代码。
```

## 文档目录约定

```
docs/
├── architecture.md         # 项目架构总览（长期维护）
├── api.md                  # 接口定义汇总（长期维护）
├── changelog.md            # 改动日志（持续追加）
├── modules/[module-a].md   # 大模块说明（按需新增）
├── modules/[module-b].md   # 大模块说明（按需新增）
└── ...
```

## 最佳实践

1. **初始分析只做一次** — architecture.md 和 api.md 是基准，后续只增量更新
2. **每次提交前更新 changelog** — 保证改动记录不遗漏
3. **大模块才独立文档** — 不要为每次小改动新建 MD
4. **新任务必带文档路径** — 避免 AI 盲目分析浪费时间
5. **定期清理过时文档** — 重构后删除或更新旧文档
